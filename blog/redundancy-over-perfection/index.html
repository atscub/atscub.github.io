We aren't as intelligent as we think

Every time someone says “AI hallucination,” my first thought is: yeah, because humans don’t hallucinate all the time.

Think about it. Have you ever argued so strongly about something, totally convinced that you were right, only to find out later that you were embarrassingly wrong? You just imagined the whole thing so vividly you thought it was true.

What “AI hallucination” really means is non-verified imagination. The model generates something without enough structure to verify it. That’s the whole problem. It is not broken, it’s just missing something.

But the conversation never goes there. Instead it goes to: “See? AI makes things up. Can’t be trusted.”

And I get it, I used to be that person, it is reassuring, it appeases the fears of being “replaced”. But I think that framing is dangerous in many ways because it hinders us from really understanding what’s coming and how to prepare. And honestly we are not that perfect ourselves and this kind of reaction proves it.

Humans are LLMs with better structure and better sensors.

LLMs are more similar to a biological brain than is comfortable for us to admit.

We call our version of it experience, instinct, training. That last one sounds familiar.

When a senior developer glances at a code diff and immediately says “this will break under concurrency” — that’s not formal proof. That’s inference from patterns they’ve seen before.

We call it intuition because it sounds better than “lossy probabilistic inference based on a biased training set.” But that’s what it is.

In fact, many IQ tests are just pattern matching. “Look at this sequence. What comes next?” That’s what we decided intelligence is.

But when an AI does it, suddenly it’s “just” statistics. It doesn’t really understand.

I don’t know. Maybe. But I’d be more convinced by that argument if I hadn’t watched my wife swear she locked the front door, argue about it for ten minutes, go back to check, find it wide open, and then say “well it was locked when I checked.” We are not that consistent ourselves.

There is an interesting work in psychology by Daniel Kahneman, that proposed that the brain has a fast mode and a slow mode. System 1 and System 2. One is instant pattern matching, the other is step-by-step deliberation.

But I think System 2 is just System 1 in a loop. Every step in that “deep” structured thinking is still an inference. As the task complexity decreases, inference becomes more efficient than logic, things are obvious, 6x7=42, you don’t reason about every time, you just remember it. In fact you can not reason ad infinitum, we have to set some baseline, that’s what we mathematicians call axioms, things that we simply assume are true, without proof. Complex thought is therefore, a combination of inference and logic.

That’s exactly what a context window and a chain of thought attempt to replicate.

We’re not as good as we think. We’re just well-scaffolded.

Have you ever met anyone who got a non-trivial program running on the first compilation?

Not a hello world. A real feature with business logic and edge cases. Has anyone, ever, one-shot that with zero errors?

I haven’t. Not once. And I’ve been doing this for a while.

But nobody walks out of that experience saying “developers hallucinate.” Nobody says “human cognition can’t be trusted” because you had a bug on line 47. We just… fix it. Recompile. Move on. The entire development process is designed around the assumption that you will get it wrong on the first try. Probably on the second try too.

The whole infrastructure of modern software — code review, tests, CI pipelines, QA — is a monument to the fact that no single person gets it right alone. We screw up, a lot, but we iterate until the mess is clean.

Nobody finds this controversial.

But when an AI model makes a mistake? Suddenly the technology is DOA. Shoot down the servers and let’s go back to writing code the good old way.

We look at our own outputs — the polished, reviewed, tested, iterated final product — and compare that to a model’s raw first draft. Then we conclude we’re better. But our reliability doesn’t come from some innate perfection in our individual reasoning. It comes from the process. Remove all that. Put a developer alone with no tests, no review, no feedback, deploying straight to production. See how reliable that is.

So if we’re both imperfect pattern-matchers, and we already know that process is what makes us reliable — why wouldn’t the same work for AI?

Redundancy = reliability

A single node with a 5% failure rate fails 5% of the time. Add a second independent node checking the same thing. Now both need to fail for the system to fail. That’s 0.25%. A third node? 0.0125%.

This is not clever. This is multiplication. And it’s the principle behind every reliable system ever built — RAID arrays, aviation checklists, the judicial system using twelve imperfect biased emotional humans instead of trusting one really smart judge.

The world is imperfect. Resilient systems rely on redundancy.

At this point, it doesn’t matter if the models get better. They are already good enough for redundancy to close the gap. Fire has been discovered.

I tested this

When working on Food Fighters a week ago, the whole thing was autonomous — Claude running subagents that handled different parts of the process.

It was messy but I was trying to prove a point. That agents in a loop with several of them checking on each other would allow for reliable improvement and progression, and though it was a duct-tape monster, it worked.

For all of that to fail at the same time? For the code writer to introduce a bug, and the reviewer to miss it, and the tests to not catch it, and the type checker to not flag it, and the visual validator to not see it?

At that point, you really need to ask, would I have done it better anyway?

Any single node working correctly is enough to self-correct the system. One functioning check in the chain is enough.

During the best phase of that project, the AI pushed 18 commits in two hours, completely on autopilot. Screen shake, hit flash, floating damage numbers, combo counter, particle effects. It wasn’t better because it didn’t have better tooling. Assessing something as dynamic as a game through screenshots is not very efficient.

I use the same pattern at work. Not for games — for production code. Multiple adversarial agents. One writes. Others review. Plus good old deterministic checks: lint, tests, typechecks. The whole thing is set up so that no single agent’s mistake can survive the pipeline unchallenged.

And it works. No individual node is reliable, but the whole system is.

But if you won’t listen to me, Stripe is doing it at a massive scale.

As Steve Yegge says, “it is all a loop”.
